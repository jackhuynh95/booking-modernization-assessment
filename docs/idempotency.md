# Idempotency And Eventual Consistency

## Overview

This document defines retry-safe command handling, outbox/inbox messaging, eventual consistency, reconciliation, and edge-case behavior for the proposed booking microservice.

The design assumes an Expedia-like travel platform. The booking service owns booking lifecycle state and orchestration, while payment, inventory, traveller profile, supplier, communications, legacy database, and reporting/search systems remain separate dependencies.

## Goals

- Make booking commands safe for client, worker, and gateway retries.
- Prevent duplicate bookings, duplicate confirmations, duplicate cancellations, and duplicate expiry actions.
- Publish integration events reliably without distributed transactions.
- Let payment, inventory, legacy sync, communications, and projections converge through events and reconciliation.
- Expose clear pending, terminal, retryable, and conflict states to callers.

## Command Idempotency Strategy

All mutating booking commands require an `Idempotency-Key` header. The key is generated by the caller or trusted worker and is scoped by caller identity, endpoint/command type, and booking ID where applicable.

Required commands:

| Command | Endpoint | Key scope | Reason |
| --- | --- | --- | --- |
| Create booking | `POST /api/bookings` | Caller + `CreateBooking` + key | Prevent duplicate booking records when web/mobile clients retry after timeout. |
| Confirm booking | `POST /api/bookings/{bookingId}/confirm` | Caller/worker + `ConfirmBooking` + booking ID + key | Prevent duplicate confirmation transitions, payment captures, or supplier confirmation attempts. |
| Cancel booking | `POST /api/bookings/{bookingId}/cancel` | Caller/worker + `CancelBooking` + booking ID + key | Prevent duplicate cancellation, refund, inventory release, and notification side effects. |
| Expire booking | `POST /api/bookings/{bookingId}/expire` | Worker + `ExpireBooking` + booking ID + key | Prevent concurrent expiry workers from emitting duplicate logical expiry events. |

The API rejects missing idempotency keys on mutating endpoints with `400 Bad Request`. Keys should have enough entropy to avoid accidental collision, for example UUID/ULID values.

## Idempotency Record

The idempotency store is service-owned persistence, normally in the booking service database. It should be written inside the same application transaction used for booking state changes where a command changes state.

| Field | Purpose |
| --- | --- |
| `key` | Original `Idempotency-Key`, scoped by caller, command type, and booking ID when applicable. |
| `requestHash` | Canonical hash of method, route, command type, significant request body fields, and caller scope. Excludes volatile headers such as correlation ID. |
| `commandType` | `CreateBooking`, `ConfirmBooking`, `CancelBooking`, or `ExpireBooking`. |
| `bookingId` | Service-owned booking ID. For create, populated after the booking ID is allocated. |
| `responseSnapshot` | Stored HTTP status and response body returned for the original command result or accepted state. |
| `status` | `InProgress`, `Completed`, `FailedRetryable`, `FailedTerminal`, or `Conflict`. |
| `expiry` | Retention timestamp. Keep long enough to cover client retries, payment/inventory retries, and operational replay windows. |

Operational metadata should also include `createdAt`, `updatedAt`, `correlationId`, `callerId`, and failure reason, but those are support fields rather than the core contract.

## Duplicate Request Behavior

| Replay condition | Behavior | API response |
| --- | --- | --- |
| Same key, same command type, same caller scope, same request hash | Return stored `responseSnapshot`. Do not re-run domain transition or emit a new logical event. | Original `201 Created`, `202 Accepted`, `200 OK`, or stored terminal error. |
| Same key, different request hash | Reject as unsafe key reuse. Do not mutate booking state. | `409 Conflict` with Problem Details code such as `idempotency_key_payload_mismatch`. |
| Same key, different command type or booking scope | Reject as unsafe key reuse. | `409 Conflict`. |
| Key exists with `InProgress` | Return current accepted/pending representation or `409 Conflict` depending on whether command ownership is still active. Prefer `202 Accepted` for known in-flight commands. | `202 Accepted` with booking status when safe. |
| Key expired and record purged | Treat as new command. Business rules and booking state still prevent invalid duplicate transitions. | Depends on current state. |

The request hash must use a canonical JSON representation so harmless field ordering differences do not create false conflicts.

## State And Concurrency Controls

Idempotency prevents repeated command execution for the same key. It does not replace state transition rules or optimistic concurrency.

- Booking aggregate has a monotonic `version`.
- Command requests may include `expectedVersion`.
- Confirm, cancel, and expire load current booking state and apply allowed transition rules.
- Terminal states `Confirmed`, `Cancelled`, `Failed`, and `Expired` are not overwritten by late or duplicate commands except through an explicit administrative correction workflow outside this assessment scope.
- Concurrent confirm/cancel/expire commands use optimistic concurrency; only one state transition commits for a given aggregate version.

## Transactional Outbox Pattern

The booking service uses a transactional outbox because the database and message broker cannot commit atomically.

1. Command handler validates idempotency and loads the booking aggregate.
2. Domain transition is applied, for example `PendingPayment` to `Confirmed`.
3. Booking state, idempotency record, and outbox event are saved in the same database transaction.
4. Outbox record contains stable `eventId`, `eventType`, `aggregateId`, `aggregateVersion`, payload, `correlationId`, status, retry count, and next attempt time.
5. Outbox worker publishes asynchronously to the broker after transaction commit.
6. Worker marks the outbox record as published only after broker acknowledgement.
7. Transient failures retry with exponential backoff and jitter.
8. Repeated failures move to `DeadLettered` or `RequiresOperationalReview` with alerting, correlation ID, failure reason, and replay instructions.

If publish succeeds but broker acknowledgement is lost, the outbox worker may publish the same event again with the same `eventId`. Consumers must dedupe by `eventId`.

## Inbox And Consumer Dedupe

All consumers of booking events, including communications, reporting/search, legacy sync, payment workflow, inventory workflow, and trip projections, should treat delivery as at least once.

Consumer expectations:

- Store inbound `eventId` in an inbox/dedupe table before or inside the same transaction as consumer side effects.
- Ignore duplicate `eventId` deliveries.
- Store `aggregateId`, `aggregateVersion`, `eventType`, `schemaVersion`, `correlationId`, processed timestamp, and handler result.
- Reject stale aggregate versions for projections unless the consumer is in explicit replay/rebuild mode.
- Make external side effects idempotent by natural business key, for example email send by `eventId`, refund by `bookingId` + cancellation version, or legacy write by `bookingId` + aggregate version.
- Dead-letter poison messages after bounded retries and alert the owning team.

## Eventual Consistency Model

The booking API shows the canonical booking state for migrated flows. Other systems converge asynchronously.

| Dependency | Consistency approach | Booking behavior |
| --- | --- | --- |
| Payment | Payment authorization/settlement remains external. Booking stores payment references and waits for `PaymentAuthorised` or `PaymentFailed`. | Use `PendingPayment` until outcome. Confirm only after required payment outcome. Trigger refund/void workflow after cancellation when payment already succeeded. |
| Inventory | Inventory/availability remains source of truth for holds and capacity. Booking stores hold refs and line outcomes. | Use `Requested` or `PendingPayment` while hold converts. Fail or offer alternate handling if `InventoryUnavailable` arrives. Release hold after cancellation/expiry. |
| Traveller | Traveller profile owns identity and mutable profile fields. Booking stores stable `travellerId`, passenger refs, and contact snapshot needed for audit/communications. | Validate traveller at command time. Later profile changes do not rewrite historical booking audit fields. |
| Legacy DB sync | Booking service is canonical for migrated flows; legacy DB is transitional read/write compatibility. | Legacy sync runs from events/outbox. Sync failures mark `legacyReference.migrationStatus` as `sync_pending` or `sync_failed`, not rollback confirmed booking blindly. |
| Communications | Email/SMS/push is event-driven. | Communications consumes `BookingConfirmed`, `BookingCancelled`, `BookingFailed`, and `BookingExpired`; duplicate events must not send duplicate messages. |
| Reporting/search projections | Projections consume events and may lag. | GET by booking ID uses service-owned state; search/reporting views are eventually consistent and track last processed aggregate version. |

## No Distributed Transactions

The design intentionally avoids distributed transactions across booking, payment, inventory, supplier, legacy DB, broker, communications, and reporting/search.

Reasons:

- These systems have different data stores, failure modes, owners, and latency profiles.
- Travel booking flows naturally include pending states, supplier delays, payment windows, and inventory hold expiry.
- Two-phase commit would increase coupling and still would not make downstream side effects such as emails or supplier calls truly atomic.

Instead, the design uses explicit booking states, idempotent commands, transactional outbox, consumer inbox/dedupe, compensating actions, and reconciliation jobs.

## Reconciliation Strategy

Reconciliation workers compare durable system-of-record views and create repair tasks or compensating commands. They should be read-only by default and only mutate through normal idempotent commands.

| Reconciliation | Check | Repair path |
| --- | --- | --- |
| Booking store vs legacy DB | For migrated bookings, compare `bookingId`, optional `legacyBookingId`, status, totals, line refs, and migration status. | Re-run legacy sync idempotently, mark `sync_failed`, or create operational review item. Booking service remains canonical. |
| Booking vs payment | Compare booking status/payment reference against payment authorization, failure, void, refund, and settlement state. | If payment succeeded but booking failed/cancelled, trigger refund/void workflow. If booking awaits payment and payment exists, consume/replay payment outcome. |
| Booking vs inventory hold | Compare booking line status, hold reference, hold expiry, supplier confirmation, and inventory state. | Release stale holds for cancelled/expired bookings. Fail pending bookings with unavailable/expired holds. Re-request confirmation only when business rules allow. |
| Outbox published state vs broker | Compare outbox `Published` records, broker/audit telemetry, and consumer lag/dead-letter queues. | Republish unpublished/unknown records with same `eventId`; investigate dead-lettered records; rebuild projections from outbox/event history. |

Reconciliation output should include `correlationId`, impacted `bookingId`, current states, proposed action, owner, and severity.

## Edge Cases

### Duplicate Booking Submission

| Item | Behavior |
| --- | --- |
| Scenario | Web/mobile client times out after `POST /api/bookings` and retries with the same `Idempotency-Key`. |
| Expected behavior | Booking service returns the original response snapshot. No second booking is created. |
| State transition | Original transition remains `none -> Requested` or `none -> PendingPayment`; replay has no transition. |
| Event impact | Original `BookingRequested` outbox event remains the only logical event. Broker duplicate publish is still possible with same `eventId`. |
| Retry/reconciliation | Client can retry safely. Reconciliation checks duplicate traveller/quote/line patterns only as fraud/support signal, not normal flow. |
| User/API response | Original `201 Created` or `202 Accepted` with same `bookingId`. |

### Same Idempotency Key With Different Payload

| Item | Behavior |
| --- | --- |
| Scenario | Caller reuses a key but changes traveller, booking lines, payment reference, cancellation reason, or expected version. |
| Expected behavior | Reject request. Key reuse is ambiguous and unsafe. |
| State transition | No booking state change. |
| Event impact | No new outbox event. |
| Retry/reconciliation | Caller must send a new idempotency key for a genuinely new command. Security/ops can inspect repeated conflicts. |
| User/API response | `409 Conflict` with `idempotency_key_payload_mismatch`. |

### Payment Succeeds But Booking Confirmation Fails

| Item | Behavior |
| --- | --- |
| Scenario | Payment gateway authorizes payment, but confirm command fails because inventory/supplier confirmation fails or booking state changed. |
| Expected behavior | Booking does not pretend confirmation succeeded. It moves to `Failed` if non-recoverable or stays pending if retryable. Payment compensation starts. |
| State transition | `PendingPayment -> Failed` for non-recoverable inventory/supplier failure, or remains `PendingPayment` with retryable failure metadata. |
| Event impact | Emit `BookingFailed` when terminal. Payment/refund workflow consumes failure or cancellation event and voids/refunds by idempotent payment reference. |
| Retry/reconciliation | Booking/payment reconciliation detects authorised payment without confirmed booking and triggers refund/void or manual review. |
| User/API response | Confirm command returns `202 Accepted` while workflow continues, or `422`/`409` if command cannot be accepted. GET shows `Failed` or `PendingPayment`. |

### Inventory Unavailable After Booking Request

| Item | Behavior |
| --- | --- |
| Scenario | Booking request is accepted, then inventory service reports no capacity or hold expired before confirmation. |
| Expected behavior | Booking fails or remains pending only if alternate line handling is supported. For this assessment, default is fail terminally. |
| State transition | `Requested` or `PendingPayment -> Failed` with `failureCode = inventory_unavailable`. |
| Event impact | Emit `BookingFailed`. Inventory release is unnecessary if no hold exists; payment workflow voids/refunds any payment. Communications can notify traveller. |
| Retry/reconciliation | Inventory reconciliation confirms no active hold remains. User may create a new booking with a fresh quote and idempotency key. |
| User/API response | GET returns `Failed`; create/confirm may have returned `202 Accepted` earlier because outcome was asynchronous. |

### Cancellation While Payment Is Pending

| Item | Behavior |
| --- | --- |
| Scenario | Traveller cancels while booking is `PendingPayment` and payment outcome has not arrived. |
| Expected behavior | Cancellation wins if policy allows. Payment workflow is told to stop, void, or refund if a late authorization appears. |
| State transition | `PendingPayment -> Cancelled`. |
| Event impact | Emit `BookingCancelled` with `refundExpected` based on payment state. Late `PaymentAuthorised` is treated as compensation input, not confirmation permission. |
| Retry/reconciliation | Payment reconciliation looks for late authorization against cancelled booking and triggers refund/void. Inventory release runs idempotently. |
| User/API response | `202 Accepted` or `200 OK` with `status = Cancelled`; GET may show refund/legacy sync still pending. |

### Legacy DB Timeout Or Sync Failure

| Item | Behavior |
| --- | --- |
| Scenario | Booking state commits in service DB but legacy adapter times out or rejects sync write. |
| Expected behavior | Do not roll back service-owned booking state for migrated flow. Mark legacy sync pending/failed and alert/retry. |
| State transition | Booking lifecycle state remains as committed, for example `Confirmed`; migration status changes to `sync_pending` or `sync_failed`. |
| Event impact | Lifecycle event remains published through outbox. Legacy sync consumer may dead-letter or create operational review item. |
| Retry/reconciliation | Legacy sync worker retries with backoff using idempotent legacy write key. Reconciliation compares service store and legacy DB. |
| User/API response | Command can return `202 Accepted`; GET shows booking status plus optional `legacyReference.migrationStatus`. |

### Outbox Publish Succeeds But Acknowledgement Is Lost

| Item | Behavior |
| --- | --- |
| Scenario | Broker accepts `BookingConfirmed`, but worker times out before acknowledgement and treats publish as unknown. |
| Expected behavior | Worker republishes same outbox record with same `eventId`. |
| State transition | No booking state change. |
| Event impact | Consumers may receive duplicate message; inbox dedupe ignores second delivery. |
| Retry/reconciliation | Outbox remains retryable until acknowledgement or operational review. Broker/consumer telemetry helps confirm delivery. |
| User/API response | No direct API change. GET returns committed booking state. |

### Event Replay Or Duplicate Event Delivery

| Item | Behavior |
| --- | --- |
| Scenario | Broker redelivers an event, or operations replay events to rebuild reporting/search/trip projections. |
| Expected behavior | Consumers dedupe by `eventId`; replay mode can rebuild from ordered aggregate versions. |
| State transition | Booking service does not change state from outbound replay. Consumer projection may update to current event version. |
| Event impact | No new logical event. Replayed event keeps original `eventId`, `occurredAt`, and aggregate version. |
| Retry/reconciliation | Projection stores last processed aggregate version and can detect gaps. Dead-lettered events can be replayed after fix. |
| User/API response | Search/reporting may lag during replay; booking detail endpoint remains canonical. |

### Late Payment Or Inventory Event After Cancellation Or Expiry

| Item | Behavior |
| --- | --- |
| Scenario | `PaymentAuthorised`, `InventoryHeld`, or `SupplierConfirmed` arrives after booking is `Cancelled` or `Expired`. |
| Expected behavior | Do not resurrect booking. Treat late event as compensation input. |
| State transition | Terminal state remains `Cancelled` or `Expired`. Compensation metadata may be recorded. |
| Event impact | No `BookingConfirmed` is emitted. Refund/void or inventory release workflow may emit its own downstream events outside booking lifecycle. |
| Retry/reconciliation | Reconciliation checks late payment/hold against terminal booking and triggers refund, void, release, or manual review. |
| User/API response | GET remains `Cancelled` or `Expired`; support view may show compensation pending. |

### Concurrent Confirm And Cancel Commands

| Item | Behavior |
| --- | --- |
| Scenario | Confirm workflow and traveller cancellation both attempt to update the same pending booking. |
| Expected behavior | Optimistic concurrency allows only one transition to commit. The losing command reloads current state and either returns conflict or stored idempotent result. |
| State transition | Either `PendingPayment -> Confirmed` or `PendingPayment -> Cancelled`, never both for the same aggregate version. |
| Event impact | Only the committed transition inserts an outbox event. A later valid compensating action may emit a new event, for example cancellation after confirmation if policy allows. |
| Retry/reconciliation | Losing worker retries only if current state still allows its command. Reconciliation detects impossible combinations such as confirmed booking with active cancellation marker. |
| User/API response | Winner returns `200 OK`/`202 Accepted`; loser returns `409 Conflict` or `422 Unprocessable Entity` unless replaying its own completed idempotency key. |

## Phase 5 And 6 Decisions

- Idempotency keys are mandatory for create, confirm, cancel, and expire commands.
- Duplicate same-key/same-payload commands return the stored response; same-key/different-payload commands return `409 Conflict`.
- Booking state, idempotency result, and outbox event are committed together when a command changes state.
- Outbox publishing is asynchronous, retryable, and at least once; consumers use inbox/dedupe.
- Payment, inventory, traveller, communications, reporting/search, and legacy DB dependencies are eventually consistent.
- No distributed transaction is used across services.
- Reconciliation is a first-class reliability mechanism and compares booking store with legacy DB, payment, inventory, and broker/outbox state.
- Edge cases are handled through explicit state transitions, idempotent retries, duplicate-event tolerance, and compensation rather than hidden best-effort side effects.
